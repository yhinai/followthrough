## Live worker policy gate — not accepted

The live task is intentionally parked and must not be counted as successful
execution:

- `t_6ca638f8` run 2 ended with outcome `reclaimed`.
- The task is `ready` with `assignee: null`, the Hermes representation observed
  for a parked/reclaimed task.
- The recorded reclaim reason says the generic agent profile auto-started a
  broad Docker-backed MCP toolserver. That violates the mandatory no-Docker,
  least-authority Followthrough worker policy.
- Earlier task `t_8cf901d7` was also reclaimed after its worker planned Docker
  instead of the mandatory Followthrough runner.

No task may be resumed until a dedicated least-authority worker profile is in
place. The source mapping fix is tested, but the already-running orchestrator
process must be reloaded by its owner before its live database reconciliation
can use the new mapping.

## Native runner containment evidence

- The known-good fixture executed inside a transient
  `systemd-service+bubblewrap` sandbox with network disabled, a writable workspace
  only, provenance commit/tree receipts, and workspace cleanup.
- The malicious fixture was denied by static policy. Under the explicit red-team
  override, the sandbox blocked Hermes home, Docker socket, GPU, host-home and
  root writes, and network access; the host escape marker was not created.
- The timeout fixture was killed within the bounded test window and its
  workspace was removed.
- Package-install commands are refused by default, remote sources require HTTPS
  on an allowlisted forge, and common BSD license forms are identified.

These focused tests establish P4-01 and P4-02 for the native runner boundary.
They do not establish that arbitrary untrusted repositories are safe, and they
do not override the failed live generic-worker policy gate above.
